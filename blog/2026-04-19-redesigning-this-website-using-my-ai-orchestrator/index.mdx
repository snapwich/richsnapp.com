This site looks different. If you're a returning reader you've probably noticed by now. I redesigned the whole thing
using [jr](/article/2026/automating-your-agents), the AI orchestrator I built on top of Claude Code, git worktrees, and
the workflow I described in my [last article](/article/2026/automating-your-agents). If you haven't read that, the short
version: jr breaks work into features and tasks, runs AI agents in isolated worktrees with code review and architect
review at each stage, and gates for human review before merging. It's mostly autonomous, but human-in-the-loop where it
matters.

I set out to test whether an exhaustive plan plus orchestrated AI agents could handle a non-trivial redesign from end to
end. The site needed a Bootstrap 3-to-5 upgrade, a LESS-to-Sass migration, a complete layout overhaul, new components, a
homepage redesign, dark mode cleanup, and a new typography system. Not a greenfield project — a refactor of a live site
with existing content, URLs, and functionality that all needed to be preserved.

The answer: the orchestrator got me about 80% of the way there. The remaining 20% was irreducibly human. Not because the
tooling failed, but because certain categories of work fundamentally resist automation. Here's what that looked like.

## The process

I started by creating HTML mockups in Claude for the key pages — homepage, blog post, tags page. These served as visual
targets, not pixel-perfect specs. Then I wrote a detailed refactor plan covering tests, migrations, layout, and
components. jr's `/plan-features` command broke it into 11 features with 30+ tasks in dependency order — E2E tests first
as a safety net, then infrastructure migrations, then visual changes. Two more features were added after the initial
plan to address gaps. Each feature maps to a PR, each task to a commit.

What follows is a walk through the key moments of this redesign — what the orchestrator handled, where the plan fell
short, where my own taste needed to take over, and where the automated process introduced bugs I had to fix.

The concept designs were created as static HTML files in Claude's web interface. They defined the color system,
typography, component structure, and layout grid. Agents referenced specific line numbers in these files when
implementing components. These concepts were explicitly "visual guides only" — they captured what the site should look
like but couldn't capture how it should _feel_ to use.

</Step>

<Step image={imgJustTree} textPosition="left" alt="just tree output showing the full ticket hierarchy" label="The Ticket Hierarchy">

`/jr:plan-features` broke an intial planning session into 11 features with dependencies. E2E tests first, then
infrastructure (LESS→Sass, Bootstrap 3→5, dark mode cleanup), then visual changes (figures, typography, layout,
components, homepage). Two more features — Style Cleanup and Code Simplification — were added later to cover gaps.

</Step>

<Step image={imgSpecTable} alt="Components with vs without concept specs and their post-merge fix counts" label="Spec Depth = Quality">

The clearest signal from the retros: **spec completeness directly predicted output quality.** Components with complete
concept specs (Pill, PostCard, Breadcrumb, PrevNextNav) needed zero post-merge corrections. Components where the spec
covered appearance but not behavior or accessibility (ThemeToggle) or had no spec at all (ShareButtons) accounted for 7
post-merge fix commits. The ThemeToggle looked right in the topbar but broke when embedded in articles — duplicate IDs,
missing contrast, broken focus states. None of that was in the concept.

</Step>

<Step image={imgSmallMixin} textPosition="left" alt="Ticket note showing the .small() mixin discovery during LESS to Sass migration" label="Agent Discovery">

The agents made genuinely non-obvious discoveries. During the LESS-to-Sass migration, a coder found that a `.small()`
mixin call inside a `figure` block resolved to a _local_ `&.small` selector — float and width styling — not Bootstrap's
global `.small` which only sets font-size. The agent recognized the distinction, created a dedicated
`@mixin figure-small()` in Sass, and left a note on the parent ticket so sibling tasks would know. This is the kind of
careful, context-aware work the orchestrator produces when the task is well-scoped.

</Step>

<Step image={imgHomepageBefore} imageAfter={imgHomepageAfter} alt="The homepage after the orchestrator built it in 15 minutes" label="Homepage in 15 Minutes">

The homepage was a clean orchestrator win. The coder and reviewer completed 813 lines across two tasks in about 15
minutes. The data layer was cleanly extracted with 13 unit tests, and the coder independently recognized that
DefaultLayout's content wrapper would block the full-width hero and created a separate HomeLayout. Good autonomous
architectural decisions.

</Step>

<Step image={imgHomepagePrePolish} imageAfter={imgHomepageAfter} textPosition="left" alt="Homepage before and after manual polish commits" label="Homepage Polish">

But 12+ polish commits followed. Hero padding, avatar positioning, eyebrow visibility, sidebar search colors, preview
image sizing — all required manual iteration. The orchestrator built the structure; the human refined the feel.

</Step>

<Step image={imgToc} alt="Table of Contents in the sidebar and mobile hamburger menu" label="Planning Gap: The Missing ToC">

The biggest planning gap: the Table of Contents. Six commits, ~750 lines, integrated into both the sidebar and the
mobile hamburger menu. It wasn't in any plan, concept, or ticket. The original site didn't even have one. But the
redesign moved primary navigation from a sidebar running down the right margin into the topbar. All my figures had been
designed to avoid that sidebar space — and now it was empty. Something felt missing. A ToC was the obvious use for that
space, but I couldn't have planned for it because the need only emerged from seeing the finished layout. Other gaps: the
Steps component you're reading right now, swapping Font Awesome for astro-icons, and an Astro framework upgrade to use
its new font API.

</Step>

<Step image={imgFontSwap} textPosition="left" alt="Before and after comparison of Newsreader vs Noto Serif body text" label="Taste: The Font Swap">

The plan specified Newsreader for body text. The agents implemented it faithfully. Five days later, I switched back to
Noto Serif — it didn't read the way I expected at body sizes. Alongside the swap, I noticed the darker background in
dark mode made text harder to read without reducing the font weight to 300 — a common luminosity problem. I scoped the
lighter weight to article body text only. Roboto Mono needed to stay at 400 in dark mode or it looked too thin. These
are typographic nuances that emerge from sustained reading, not from a mockup.

</Step>

<Step image={imgEyebrow} alt="Avatar covering the eyebrow on desktop, and mobile view with eyebrow visible" label="Taste: The Eyebrow">

I opted for a photo avatar instead of the concept's gradient circle — I wanted the site to feel more personal. I made it
larger at desktop sizes (120px) with a scroll-driven shrink animation. But this covered the "Latest Article" eyebrow
text above the hero, so I removed it. On mobile, where the avatar was already small, the hero article lost its
distinction without the label. So I added it back at smaller breakpoints only. Two commits — the first deleted 70 lines,
the second added 70 lines — for responsive behavior that only made sense after seeing the avatar interaction at multiple
viewport sizes.

</Step>

<Step image={imgTopbarConcept} textPosition="left" alt="Concept topbar design compared to the shipped 757-line Topbar component" label="Taste: Concept vs Reality">

The concept topbar was 52 lines of CSS — a static mockup with no hamburger menu, no responsive behavior, no theme-aware
avatars. The shipped Topbar component is 757 lines. The gap between "visual guide" and "implementable spec" was
enormous, and the agents could only bridge the structural part of it. Getting the orchestrator to match the concept
exactly was painful; I went with what came out and refined from there.

</Step>

<Step image={imgFigureReview} alt="Figure system ticket notes showing 5 review rounds and architect self-diagnosis" label="Drift: Figure System Review">

The figure system was the highest-drama feature. Five review rounds before merging. The architect approved round one
with "0 significant gaps" — I immediately found four visual bugs. The architect's self-diagnosis after round three: _"I
should have been using the browser to inspect computed styles at specific breakpoints for each figure variant."_ Even
after five rounds, eight more fix commits followed post-merge — including `astro-island` wrapper elements breaking CSS
child selectors, a framework-specific interaction no plan anticipated.

</Step>

<Step image={imgThemeToggle} textPosition="left" alt="ThemeToggle showing accessibility contrast conflicting with concept design" label="Drift: Accessibility vs Design">

The ThemeToggle took three separate commits to fix. The agents restyled it per the concept, but accessibility
requirements (contrast ratios, focus states) kept conflicting with the visual design. Each fix regressed the other
direction. The root issue: the concept itself wasn't contrast-appropriate, so the agents were working toward a target
that was inherently inaccessible. The Bootstrap 3→5 upgrade had a similar pattern — three automated actors all approved
without noticing that different default breakpoint pixel values broke container widths across the entire site.

</Step>

<Step image={imgScreenshotDiff} alt="Visual regression diff output showing expected vs actual screenshots" label="Screenshot Tests">

Screenshot tests (visual regression) were consistently bad for AI automation but useful for me as a human. The agents
_ran_ them but couldn't meaningfully interpret hundreds of diffs — 280 out of 320 marked as "expected." Real regressions
hid among expected changes. For me, making a scoped change and running before/after caught regressions on pages I
wouldn't have checked manually. Same tool, different value — a human making a targeted change can evaluate a handful of
diffs meaningfully, while an agent reviewing a wholesale redesign cannot.

</Step>

</Steps>

## Takeaways

### The 80/20 split

The orchestrator produced the structural backbone of a complete site redesign — LESS-to-Sass conversion, Bootstrap
upgrade, component system, layout refactor, homepage — in a fraction of the time it would have taken manually. Where the
specs were detailed and the concepts existed, the output was genuinely good. First-pass approval on most tasks. Agents
making correct autonomous architectural decisions. Meaningful code review catching real bugs.

After the orchestrated features merged, 81 manual commits followed. They break into three categories:

**Planning gaps** (25%) exist because you can't fully specify what you want until you see what you have. The Table of
Contents wasn't in any plan because the need only became apparent from using the redesigned site. No amount of upfront
planning eliminates this class of requirement — it's inherent to creative work.

**Taste iteration** (44%) exists because design is embodied judgment. You can specify "Newsreader for body text" but you
can't spec whether it _reads well_ at body sizes in dark mode at 2am. That requires a human sitting with the output and
reacting to it.

**Bugs and drift** (31%) are the one category that could theoretically shrink. The figure system's five review rounds
happened because the automated reviewers weren't actually _looking_ at the output. As agents get better at visual
reasoning, this will shrink. But it won't disappear — the `<astro-island>` CSS interaction was invisible to any analysis
short of rendering the page.

### The role of the framework

Using Astro helped: its component model gave agents clear boundaries, and content collections provided typed schemas.
The architecture mapped naturally to jr's feature/task decomposition. It also hurt in specific ways — `<astro-island>`
wrapper elements broke CSS child selectors in figures, and scoped style compilation broke dark mode attribute selectors.
But the component model was essential. Without clear component boundaries, decomposing work into parallel-safe tasks
would have been much harder. The framework gave the orchestrator natural seams to cut along.

### Against the one-shot narrative

If you're evaluating agentic workflows against the "one-shot a full project" narrative: you can't. Not for a non-trivial
project where you want to exert heavy influence over the result. But an orchestrated workflow can get you the structural
80% with genuine quality — the kind of code you'd review and ship. And that 80% is a massive accelerator.

The time I spent on the 20% was substantial — 81 commits over several weeks of iterating. But the output was minimal in
terms of lines changed (tens or hundreds compared to the thousands the agents produced). I _chose_ to invest that time
to inject my preferred refinement into the end result. The orchestrator built the house. I chose the paint colors,
adjusted the furniture, and fixed the squeaky hinges. Both parts were necessary. Neither could have replaced the other.
